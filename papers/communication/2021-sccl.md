# Paper：SCCL

## Citation

- Authors：Microsoft Research 等
- Venue / Year：PPoPP 2021
- Title：Synthesizing Optimal Collective Algorithms
- Link：https://arxiv.org/abs/2008.08708

## 一句话总结

SCCL 将 collective communication algorithm 的设计问题形式化，并根据 topology 自动合成高效 collective。

## 问题

手写 collective algorithm 很难覆盖不同 topology 和 bandwidth / latency 约束。NCCL 等库通常依赖一组固定算法和启发式选择，未必对所有拓扑最优。

## 关键想法

把 collective decomposition、communication steps 和 topology constraints 放进 synthesis / search 问题中，自动寻找满足语义且性能较好的通信算法。

## System / Algorithm

需要重点理解：

- collective 的语义约束。
- topology-aware algorithm synthesis。
- send / receive / reduce / copy 这些 primitive。
- bandwidth 和 latency cost model。

## Experiments

- Hardware：多 GPU / 多节点拓扑。
- Workloads：all-reduce、all-gather、reduce-scatter 等 collectives。
- Baselines：NCCL 或手写算法。
- Metrics：latency、bandwidth、speedup。

## 复现计划

短期不复现 synthesis 系统，先借它建立分析框架：

- 用 4x4090 topology 画 communication graph，作为 PCIe-only 热身。
- 对比 near pair / cross-NUMA pair。
- 思考 all-reduce / all-to-all 在该 topology 上的合理 schedule。
- 用 NCCL tests 观察默认算法是否符合预期。

## 与我的研究主线的关系

SCCL 提供了 topology-aware collective optimization 的理论入口。它能帮助解释为什么同一个 collective 在不同 topology 上性能差异巨大。

## Open Questions

- 4x4090 dual-NUMA PCIe topology 上是否存在比 NCCL 默认更适合的 collective schedule？
- synthesis 类方法能否用于 LLM inference 中小 message / 高频 collectives？
- 如何把 vLLM workload 的 message size 分布放进 collective synthesis？
- 对 DeepEP V2 这类 MoE-aware runtime，collective synthesis 的思想能否迁移到 dispatch/combine path，而不只是标准 collective？
